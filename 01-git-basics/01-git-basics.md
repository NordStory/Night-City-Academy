# Урок 1. Git: восстановление ARASAKA Identity Gateway

> `Corpo Delivery Track`  
> В этом уроке ученик восстанавливает сломанный портал авторизации ARASAKA CORPORATION, учится фиксировать изменения, выпускать `hotfix`-ветки, сливать изменения, решать конфликты и безопасно откатывать неудачные правки.

## Кто ведёт урок

`Corpo Delivery Track`

Этот урок ведёт фракция `Corpo`, потому что тема требует:

- предсказуемого управления изменениями;
- дисциплины ветвления и слияния;
- понимания, как снижать риск даже в маленьком проекте.

## Зачем это нужно в реальной работе

Git для инфраструктурного инженера — это не просто способ хранить файлы. Через Git проходят:

- конфигурация;
- пайплайны;
- IaC;
- hotfix-изменения;
- review и rollback;
- выпуск правок в production.

Если инженер умеет только `git add .` и `git push`, он легко создаёт риск для команды. Если инженер понимает историю, ветки, конфликты и откат, он делает изменения управляемыми.

## Проект урока

В качестве учебного проекта используется статический сайт:

- `index.html` — портал авторизации `ARASAKA CORPORATION`;

Сценарий урока:

- по ходу урока ты пройдёшь базовый жизненный цикл изменения: commit -> hotfix -> merge -> conflict -> rebase -> remote -> rollback.

## Краткая теория

Git хранит историю проекта как последовательность коммитов. Каждый коммит — это сохранённое состояние файлов в определённый момент времени.

У Git есть три основные области:

- **working tree** — файлы, которые ты редактируешь прямо сейчас;
- **staging area / index** — изменения, которые войдут в следующий коммит;
- **repository history** — уже сохранённая история коммитов.

Базовый цикл работы:

1. изменить файлы;
2. проверить состояние;
3. добавить нужные изменения в индекс;
4. сделать коммит;
5. при необходимости вынести правку в отдельную ветку;
6. безопасно объединить её с основной веткой.

> `Corpo Rule:` изменение считается управляемым только тогда, когда понятно, что именно вошло в коммит, из какой ветки оно пришло и как его откатить.

## Модуль 1. Базовый цикл Git

### Что отрабатываем

Ты фиксируешь стартовое состояние сломанного портала и делаешь первую осмысленную правку в интерфейсе.

### Команды

```bash
git --version
# Проверяет, установлен ли Git и какая версия используется.

git config --global user.name "Ваше Имя"
# Задаёт имя автора коммитов для текущего пользователя.
# Флаг --global означает: применять настройку ко всем локальным репозиториям на этом компьютере.

git config --global user.email "you@example.com"
# Задаёт email автора коммитов.
# Флаг --global работает так же: настройка будет общей.

git init
# Инициализирует Git-репозиторий в текущей папке.

git status
# Показывает состояние репозитория:
# какие файлы изменены, какие ещё не отслеживаются, какие уже подготовлены к коммиту.

git add index.html style.css script.js account.html
# Добавляет указанные файлы в staging area.
# Именно эти версии файлов попадут в следующий коммит.

git commit -m "Initial Arasaka identity gateway"
# Создаёт коммит из изменений, находящихся в индексе.
# Флаг -m позволяет передать сообщение коммита прямо в команде.

git log --graph --oneline --all --decorate
# --graph — рисует текстовую схему ветвлений и слияний слева от коммитов (те самые линии и звездочки).
# --oneline — сокращает каждый коммит до одной строки (хэш + заголовок), чтобы история была компактной.
# --all — показывает все ветки сразу (не только ту, в которой ты сейчас находишься).
# --decorate — подсвечивает названия веток и теги рядом с соответствующими коммитами.

chmod +x bootstrap-employee.sh
# Делает shell-скрипт исполняемым.
# Флаг +x добавляет право на выполнение файла.

./bootstrap-employee.sh
# Запускает локальный shell-скрипт из текущей директории.
# Префикс ./ означает: взять файл из текущей папки.
```

## Дополнительный блок. Откат изменений

### Зачем это нужно

В реальной работе важно не только вносить изменения, но и быстро отменять:

- неудачную локальную правку;
- ошибочно подготовленный файл;
- плохой коммит.

### Команды отката

```bash
git diff
# Показывает разницу между текущими локальными изменениями и последним сохранённым состоянием.

git restore index.html
# Отменяет несохранённые изменения в указанном файле в working tree.
# Используй, если файл изменён локально, но ты ещё не хочешь сохранять эту правку.

git restore --staged index.html
# Убирает файл из staging area, но оставляет локальные изменения в working tree.
# Флаг --staged означает: работать именно с индексом, а не с рабочей копией.

git revert <commit_hash>
# Создаёт новый коммит, который отменяет изменения указанного коммита.
# Это безопасный способ отката уже сохранённого изменения.

git reset --soft HEAD~1
# Сдвигает HEAD на один коммит назад, но оставляет изменения в индексе.
# Флаг --soft означает мягкий откат без удаления содержимого из файлов.
# Используй только если понимаешь, зачем переписываешь локальную историю.
```

### Практическая миссия

Во время ночной смены оператор случайно внёс шумную правку в портал. Тебе нужно отменить её без потери контроля над историей.

### Шаги

1. В `index.html` временно испорть название бренда:

- было: `ARASAKA CORPORATION`
- станет: `MILITECH CORPORATION`

2. Посмотри различия:

```bash
git diff
```

3. Отмени локальную ошибку:

```bash
git restore index.html
```

4. Теперь в `account.html` измени значение `Employee Profile Vault`, затем подготовь файл к коммиту:

```bash
git add account.html
```

5. Передумай включать файл в коммит и убери его из staging area:

```bash
git restore --staged account.html
```

6. Сделай отдельный тренировочный commit с изменением в `account.html`, например измени подпись `Learning progress` на `Progress overview`, затем закоммить.

7. Найди hash этого коммита:

```bash
git log --oneline
```

8. Отмени его безопасно:

```bash
git revert <commit_hash>
```

> Откроется стандартное сообщение текстового редактора (обычно Vim), которое появляется в Git при слиянии веток.
> Система создает специальный «merge commit» и просит вас подтвердить или отредактировать заголовок этого события. Текст Merge branch... — это заголовок по умолчанию.
>
> - Строки с решеткой # — это просто подсказки, они не попадут в историю.
> - ~ — пустые строки в редакторе.
>   Как из этого выйти и продолжить?

> Если вас устраивает стандартное сообщение (обычно так и есть), нужно просто сохранить файл и выйти из редактора:
>
> - Убедитесь, что раскладка клавиатуры — английская.
> - Нажмите клавишу Esc.
> - Введите сочетание :wq (пишется внизу слева) и нажмите Enter.
> - w (write) — сохранить.
> - q (quit) — выйти.

> Чтобы больше не видеть Vim и редактировать сообщения коммитов в привычном VS Code, выполните следующие шаги:
>
> Введите эту команду в вашем терминале (Git Bash, Terminal или CMD):
> `git config --global core.editor "code --wait"

### Ожидаемый результат

- ты умеешь отменять локальные незакоммиченные правки;
- умеешь убирать файл из staging area без потери локальных изменений;
- понимаешь разницу между `restore`, `revert` и `reset`;
- умеешь безопасно отменять уже сохранённый коммит.

## Модуль 2. Hotfix-ветка и merge

### Что отрабатываем

Ты восстанавливаешь сломанный портал авторизации через `hotfix`-ветку и объединяешь изменение с основной линией разработки.

### Команды

```bash
git branch
# Показывает локальные ветки.

git switch -c hotfix/restore-auth
# Создаёт новую ветку и сразу переключает в неё.
# switch отвечает за переключение между ветками.
# Флаг -c означает create: создать новую ветку.
# Ветка создаётся от текущей ветки и текущего коммита.

git switch master
# Переключает на ветку master.
# Если у тебя основная ветка называется main, используй main.

git merge hotfix/restore-auth
# Вливает изменения из ветки hotfix/restore-auth в текущую ветку.

git branch -d hotfix/restore-auth
# Удаляет локальную ветку после успешного merge.
# Флаг -d означает delete.
```

### Почему важно, из какой ветки делать `switch -c`

Новая ветка создаётся от текущего состояния текущей ветки.

Если ты создашь `hotfix` не от основной ветки, а от случайной feature-ветки, то:

- в hotfix попадут лишние коммиты;
- история станет грязнее;
- потом будет сложнее понимать, что именно реально исправляло инцидент.

Практическое правило:

**перед `git switch -c ...` всегда отвечай себе на вопрос: от какой ветки я хочу начать это изменение?**

### Практическая миссия

Портал не пропускает сотрудников в кабинет. Нужно выпустить контролируемый `hotfix`, который восстановит вход и исправит редирект на страницу аккаунта.

### Шаги

> Открой в браузере index.html и попробуй залогиниться, не работает? Давай исправлять:

1. Убедись, что ты находишься в основной ветке:

```bash
git switch master
git status
```

Если основная ветка называется `main`, используй `main`.

2. Создай hotfix-ветку:

```bash
git switch -c hotfix/restore-auth
```

3. В `script.js` исправь две проблемы:

- было:

```js
const authGatewayOnline = false
const redirectPath = 'acount.html'
```

- станет:

```js
const authGatewayOnline = true
const redirectPath = 'account.html'
```

4. Зафиксируй hotfix:

```bash
git add script.js
git commit -m "Restore authorization gateway"
```

5. Вернись в основную ветку и выполни слияние:

```bash
git switch master
git merge hotfix/restore-auth
git branch -d hotfix/restore-auth
```

### Ожидаемый результат

- через отдельную ветку ты восстановил вход;
- редирект теперь ведёт в `account.html`;
- после merge основная ветка содержит рабочий hotfix;
- история отражает отдельный этап восстановления сервиса.

> `Release Policy:` hotfix должен быть минимальным. Сначала восстанавливай работоспособность, а не пытайся в той же ветке улучшить всё подряд.

## Модуль 3. Конфликты и их разрешение

### Что отрабатываем

Ты специально создаёшь конфликт в `script.js` и учишься завершать merge правильно.

### Команды

```bash
git switch -c hotfix/error-message
# Создаёт ветку для первой правки.

git switch -c feature/error-message-hints
# Создаёт вторую ветку с альтернативной правкой.

git merge hotfix/error-message
# Вливает первую ветку.

git merge feature/error-message-hints
# При попытке второго merge может возникнуть конфликт.

git status
# Показывает конфликтующие файлы и текущее состояние merge.

git add script.js
# После ручного исправления помечает файл как resolved.

git commit -m "Resolve script message conflict"
# Завершает merge после разрешения конфликта.

git merge --abort
# Отменяет текущий merge и возвращает состояние до его начала.
```

### Практическая миссия

Две команды одновременно правят текст ошибки авторизации. Нужно пройти controlled conflict и выбрать итоговую версию осознанно.

### Шаги

1. Создай первую ветку:

```bash
git switch master
git switch -c hotfix/error-message
```

2. В `script.js` измени строку `messages.invalid`:

- было:

```js
'Credentials rejected by the Arasaka identity core. Re-check the operator record.'
```

- станет:

```js
'Credentials rejected. Verify the corporate email and access token.'
```

3. Закоммить:

```bash
git add script.js
git commit -m "Clarify invalid credentials message"
```

4. Вернись в основную ветку и создай вторую ветку:

```bash
git switch master
git switch -c feature/error-message-hints
```

5. Измени ту же строку иначе:

```js
'Credentials rejected by the identity core. Contact security if the issue persists.'
```

6. Закоммить:

```bash
git add script.js
git commit -m "Add security hint to invalid message"
```

7. Влей первую ветку:

```bash
git switch master
git merge hotfix/error-message
```

8. Попробуй влить вторую:

```bash
git merge feature/error-message-hints
```

### Как выглядит конфликт

В `script.js` Git покажет конфликтные маркеры:

```js
<<<<<<< HEAD
'Credentials rejected. Verify the corporate email and access token.'
=======
'Credentials rejected by the identity core. Contact security if the issue persists.'
>>>>>>> feature/error-message-hints
```

### Как решить в VS Code

1. Открой `script.js`.
2. Найди конфликтующий блок.
3. Выбери:
   - `Accept Current Change`
   - `Accept Incoming Change`
   - `Accept Both Changes`
   - `Compare Changes`
4. Оставь итоговую версию, например:

```js
'Credentials rejected. Verify the corporate email and access token, then contact security if the issue persists.'
```

5. Заверши merge:

```bash
git add script.js
git commit -m "Resolve script message conflict"
```

### Как решить без VS Code

1. Открой `script.js` в любом редакторе.
2. Найди маркеры:
   - `<<<<<<<`
   - `=======`
   - `>>>>>>>`
3. Удали маркеры и оставь только финальную корректную строку.
4. Сохрани файл.
5. Выполни:

```bash
git add script.js
git commit -m "Resolve script message conflict"
```

### Ожидаемый результат

- ты понимаешь, как возникает конфликт;
- умеешь читать конфликтные маркеры;
- умеешь завершать конфликт через VS Code и вручную;
- понимаешь, что итоговая версия файла — это твоя осознанная ответственность.

> `Corpo Rule:` конфликт — это не сбой Git. Это момент, когда система просит человека явно подтвердить, какая версия становится корпоративной истиной.

## Модуль 4. Git Flow, `rebase` и развитие account page

### Что отрабатываем

После hotfix-починки портал уже работает. Теперь нужно доработать `account.html` в feature-ветке, синхронизировать её с `develop` и аккуратно влить изменения.

### Команды

```bash
git switch -c develop
# Создаёт ветку develop от текущей ветки.
# Поэтому перед этой командой важно стоять в актуальной стабильной ветке.

git switch -c feature/account-security-panel
# Создаёт feature-ветку для новой функциональности.
# В Git Flow такие ветки обычно ответвляются от develop.

git rebase develop
# Переносит текущую ветку поверх актуального состояния develop.
# Git берёт коммиты feature-ветки, временно убирает их,
# обновляет базу до develop и применяет эти коммиты заново.

git rebase --continue
# Продолжает rebase после разрешения конфликта.

git rebase --abort
# Отменяет текущий rebase.

git merge --no-ff feature/account-security-panel
# Выполняет merge с принудительным созданием merge-коммита.
# Флаг --no-ff означает no fast-forward.
```

### Как понимать `rebase`

`merge`:

- не переписывает историю;
- безопасен для общей ветки;
- сохраняет фактическую структуру ветвления.

`rebase`:

- переписывает историю коммитов текущей ветки;
- делает историю линейнее;
- полезен, когда ты хочешь подтянуть свежую базу из `develop` в свою личную feature-ветку;
- помогает убрать лишние merge-коммиты до финального слияния.

Когда `rebase` особенно полезен:

- ты долго работаешь в своей feature-ветке;
- `develop` уже ушёл вперёд;
- хочешь привести историю к аккуратному виду перед merge.

Когда с `rebase` надо быть осторожным:

- ветка уже расшарена и её активно используют другие люди;
- коммиты уже стали общей точкой синхронизации.

### Практическая миссия

После восстановления входа корпорация хочет расширить кабинет оператора. Нужно сделать новую панель безопасности в feature-ветке, затем встроить её поверх обновлённой `develop`.

### Шаги

1. Создай `develop`:

```bash
git switch master
git switch -c develop
```

Если основная ветка называется `main`, используй `main`.

2. Создай feature-ветку:

```bash
git switch -c feature/account-security-panel
```

3. В `account.html` после блока `<section class="profile-card" id="profile"> </section>` добавь новый блок:

```html
<section class="summary-card">
	<p class="panel-label">Security queue</p>
	<ul class="activity-list">
		<li>Pending password rotation check.</li>
		<li>Access review scheduled for sector operators.</li>
		<li>Quarterly credential audit awaiting approval.</li>
	</ul>
</section>
```

4. Закоммить:

```bash
git add account.html
git commit -m "Add security queue panel"
```

5. Вернись в `develop` и внеси параллельную правку в `account.html`:

- измени `Employee Profile Vault` на `Employee Access Vault`

6. Закоммить это в `develop`:

```bash
git add account.html
git commit -m "Refine account vault heading"
```

7. Вернись в feature-ветку и выполни `rebase`:

```bash
git switch feature/account-security-panel
git rebase develop
```

Если возникает конфликт:

```bash
git status
git add account.html
git rebase --continue
```

8. После успешного `rebase` вернись в `develop` и сделай merge:

```bash
git switch develop
git merge --no-ff feature/account-security-panel
```

9. Когда `develop` готов, перенеси его в основную ветку:

```bash
git switch master
git merge develop
```

### Ожидаемый результат

- ты попробовал `develop` + `feature/*` рабочий процесс;
- понял роль `rebase` перед финальным merge;
- доработал `account.html` через отдельную feature-ветку;
- сохранил управляемую историю изменений.

> `Release Policy:` сначала hotfix возвращает сервис в рабочее состояние. Только после этого feature-ветки расширяют функциональность.

## Модуль 5. Удалённый репозиторий, GitHub и GitHub Pages

### Что отрабатываем

Ты публикуешь урок-проект как удалённый репозиторий и подготавливаешь его к раздаче через GitHub Pages.

### Команды

```bash
git remote add origin https://github.com/USERNAME/arasaka-identity-gateway.git
# Добавляет удалённый репозиторий с именем origin.

git remote -v
# Показывает все настроенные remote-репозитории и их URL.

git branch -M main
# Переименовывает текущую ветку в main.
# Флаг -M выполняет принудительное переименование.

git push -u origin main
# Отправляет ветку main в origin.
# Флаг -u связывает локальную ветку с удалённой.

git fetch origin
# Загружает изменения с удалённого репозитория без автоматического merge.

git pull origin main
# Загружает и пытается сразу интегрировать изменения из удалённой ветки.

git clone https://github.com/USERNAME/arasaka-identity-gateway.git
# Клонирует репозиторий на другую машину.
```

### Практическая миссия

Внутренний учебный портал нужно сохранить в GitHub и развернуть как статический пример для следующих учеников.

### Шаги

1. Если основная ветка ещё `master`, переименуй её:

```bash
git switch master
git branch -M main
```

2. Создай репозиторий на GitHub, например `arasaka-identity-gateway`.
3. Подключи remote:

```bash
git remote add origin https://github.com/USERNAME/arasaka-identity-gateway.git
git remote -v
```

4. Перед публикацией внеси небольшую финальную правку, например в `account.html` добавь новый пункт в `Recent activity`:

```html
<li>Static training portal prepared for GitHub Pages deployment.</li>
```

5. Зафиксируй:

```bash
git add account.html
git commit -m "Prepare account page for publication"
```

6. Отправь проект в GitHub:

```bash
git push -u origin main
```

### GitHub Pages

Так как это статический сайт, его удобно публиковать через GitHub Pages.

1. Открой репозиторий на GitHub.
2. Перейди в `Settings`.
3. Открой `Pages`.
4. В `Build and deployment` выбери:
   - `Source` -> `Deploy from a branch`
   - `Branch` -> `main`
   - `Folder` -> `/docs`

Важно: стандартный GitHub Pages умеет публиковать либо `/root`, либо `/docs`, но не вложенную папку вида `/docs/01-git`.

Практические варианты:

- если ты публикуешь весь учебный репозиторий как GitBook-проект, используй `/docs`;
- если хочешь публиковать именно этот урок как отдельный статический сайт, удобнее вынести `index.html`, `style.css`, `script.js` и `account.html` в отдельный репозиторий или настроить GitHub Actions для деплоя нужной подпапки.

### Ожидаемый результат

- проект опубликован в GitHub;
- ты умеешь связывать локальный и удалённый репозиторий;
- статический портал можно раздать как учебный пример через GitHub Pages.

## Частые ошибки

- Ошибка: делать `hotfix` от неправильной ветки.  
  Почему это проблема: в ветку попадают лишние коммиты.  
  Как правильно: перед `git switch -c` всегда проверять текущую ветку.

- Ошибка: коммитить всё подряд без чтения `git status`.  
  Почему это проблема: в commit попадают лишние изменения.  
  Как правильно: сначала смотреть `git status`, потом выборочно делать `git add`.

- Ошибка: путать `restore`, `revert` и `reset`.  
  Почему это проблема: можно случайно переписать историю или удалить нужные изменения.  
  Как правильно: для локальных отмен использовать `restore`, для безопасного отката коммита — `revert`.

- Ошибка: решать конфликт автоматически, не читая итоговый файл.  
  Почему это проблема: можно сохранить синтаксически или смыслово неверную версию.  
  Как правильно: после конфликта обязательно вручную читать финальный файл.

- [ ] Я умею отменять локальные изменения через `git restore`.
- [ ] Я умею убирать файл из staging area через `git restore --staged`.
- [ ] Я умею безопасно откатывать плохой commit через `git revert`.
- [ ] Я умею создавать `hotfix`-ветку и вливать её через `merge`.
- [ ] Я понимаю, почему важно, из какой ветки создавать новую ветку.
- [ ] Я умею проходить merge-конфликт через VS Code и вручную.

> `Corpo Rule:` хороший Git-процесс ценен не тем, что он красив в теории, а тем, что в инциденте позволяет быстро понять: что изменили, кто изменил, где это исправить и как это откатить.
